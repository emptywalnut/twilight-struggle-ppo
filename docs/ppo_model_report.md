# Twilight Struggle PPO Model Report

This document describes the current Python PPO training stack that drives the headless Saito Twilight Struggle rules runner.

## Rules And Environment

- Rules runner: `bridge/saito_bridge.js`, controlled through JSON-lines subprocess IPC by `struggle_ai/bridge_client.py`.
- Python env: `struggle_ai/env.py`.
- RL interface:
  - `TwilightStruggleEnv`: single current-player Gymnasium env.
  - `TwilightStruggleMultiAgentEnv`: RLlib multi-agent wrapper with agent IDs `us` and `ussr`.
- Current training mode: multi-agent wrapper with side-specific policies:
  `us -> us_policy`, `ussr -> ussr_policy`.
- Action space: fixed padded `Discrete(512)`.
- Legal actions: generated by the Saito bridge at each decision; invalid padded actions are masked.
- Observation perspective: always encoded from the side-to-act perspective.

## Observation Tensors

The encoder is in `struggle_ai/features.py`.

| Tensor | Shape | Description |
| --- | ---: | --- |
| `global` | `[24]` | Side, turn, AR, VP, DEFCON, milops, space race, hand/deck/discard/removed counts, China owner, prompt phase flags. |
| `events` | `[len(EVENT_KEYS)]` | Persistent event flags and numeric event state, perspective-normalized when side-valued. |
| `countries` | `[country_count, 34]` | Per-country board state and tactical features. |
| `regions` | `[7, 20]` | Region-level aggregate scoring/control features. |
| `cards` | `[card_count, 28]` | Card metadata, public/hand-zone features, and DEFCON-risk card flags. |
| `country_adjacency` | `[country_count, country_count]` | Static map adjacency from Saito country metadata. |
| `action_mask` | `[512]` | `1` for legal action candidates, `0` for padded invalid entries. |
| `action_features` | `[512, 45]` | Structured candidate-action encoding. |
| `history_actions` | `[HISTORY_LENGTH, 45]` | Fixed-length public action-history features for transformer-history models. |
| `history_sides` | `[HISTORY_LENGTH]` | Side-to-act markers for public action history. |
| `history_turn_ar` | `[HISTORY_LENGTH, 2]` | Turn/action-round features for public action history. |
| `history_vp_defcon` | `[HISTORY_LENGTH, 2]` | VP/DEFCON features after historical public actions. |
| `history_mask` | `[HISTORY_LENGTH]` | `1` for populated history entries, `0` for padding. |

## Model Architecture

Model: `struggle_ai/rllib_masked_model.py::TorchMaskedActions`.

Default important hyperparameters:

- `hidden = 256`
- `graph_layers = 2`
- `graph_neighbor_hops = 5`
- `heuristic_prior_scale = 2.0`
- `setup_heuristic_prior_scale = 0.0` for current transfer runs because setup is forced by the heuristic path.
- `policy_temperature = configurable`
- For `--model-arch transformer_history`: `history_layers = 2`, `history_attention_heads = 4`, `history_dropout = 0.05` in the current transfer run.

Architecture:

1. Global/event encoder:
   - Concatenate `global` and `events`.
   - MLP to hidden vector.

2. Country encoder:
   - Per-country MLP from 34 country features to hidden.
   - Static adjacency plus self-loops.
   - `graph_neighbor_hops` builds a normalized multi-hop adjacency aggregation matrix.
   - For each graph layer:
     - aggregate neighbor messages with batched matrix multiply;
     - concatenate current country embedding and message;
     - transform with `Linear(2H,H) -> ReLU -> LayerNorm`;
     - residual add.
   - Mean pool all country embeddings.

3. Region encoder:
   - Per-region MLP from 20 features to hidden.
   - Mean pool all region embeddings.

4. Card encoder:
   - Per-card MLP from 28 features to hidden.
   - Mean pool all card embeddings.

5. State encoder:
   - Concatenate global, country, region, card embeddings.
   - For `transformer_history`, also encode public action history with a small transformer and concatenate its pooled embedding.
   - MLP/LayerNorm to final state embedding.

6. Action encoder and policy scoring:
   - Per-action MLP from 45 action features to hidden.
   - State query: `Linear(hidden, hidden)`.
   - Candidate logit:
     - dot product between action embedding and state query, scaled by `sqrt(hidden)`;
     - plus learned action bias;
     - plus heuristic prior feature times `heuristic_prior_scale`;
     - plus extra setup heuristic prior during setup if enabled;
     - divided by `policy_temperature`;
     - plus legal-action mask log penalty.

7. Value head:
   - MLP from state embedding to scalar value.

## Algorithm

Trainer: `struggle_ai/train_rllib.py`.

- Algorithm: RLlib PPO.
- API stack: old ModelV2 stack.
- Multi-agent mode:
  - Agents: `us`, `ussr`.
  - `us` maps to `us_policy`; `ussr` maps to `ussr_policy`.
  - Both policies use the same model class, observation features, action mask, reward shaping, environment, bridge, and PPO hyperparameters.
  - Rewards are assigned to the acting side, with zero-sum delta rewards also credited to the non-acting side when applicable.
- PPO defaults in code:
  - `gamma = 0.997`
  - `lambda_ = 0.95`
  - `clip_param = 0.2`
  - `train_batch_size`, `minibatch_size`, entropy, GPU count, and rollout settings are CLI-configurable.

## Reward Mechanisms

Implemented reward components:

| Component | Config | Behavior |
| --- | --- | --- |
| Terminal win/loss | `--terminal-reward-scale` | Win/loss from the bridge, scaled by this value. Current transfer runs use `+/-2`. |
| Nuclear-loss penalty | `--nuke-death-penalty` | Extra penalty when losing by nuclear war, thermonuclear war, or Cuban Missile Crisis. |
| VP delta | `--turn-vp-reward-scale` | Any VP change gives US `scale * vp_delta / 20`, USSR the negative. |
| High-stability coup penalty | `--high-stability-coup-penalty` | Couping stability >= 3 gives a penalty. |
| Low-stability warzone coup reward | `--low-stability-warzone-coup-reward` | Couping stability 1-2 battleground gives a reward. |
| Headline opponent event penalty | `--headline-opponent-event-penalty` | Headlining opponent non-scoring event gives a penalty. |
| DEFCON-risk card pick penalty | `--defcon-risk-pick-penalty` | Penalizes choosing a DEFCON-risk card at DEFCON 2 when the hard filter has no safe alternative and must fail open. |
| DEFCON-risk commit penalty | `--defcon-risk-commit-penalty` | Penalizes committing a DEFCON-risk event/ops path when the hard filter has no safe alternative and must fail open. |
| Empty battleground influence | `--empty-country-influence-reward` | First influence into an empty battleground gives side reward and opponent penalty. |
| Country control change | `--control-battleground-reward`, `--control-non-battleground-reward` | Old controller loses scale, new controller gains scale; `none` is ignored. |
| Max-step penalty | `--max-episode-step-penalty` | Penalizes the acting side when the episode truncates at `--max-episode-steps`. |

Current future-run intended values after recent changes:

```bash
--terminal-reward-scale 2.0
--reward-shaping-scale 1.0
--reward-shaping-final-scale 0.9
--reward-shaping-phaseout-start-episodes 100
--reward-shaping-phaseout-end-episodes 300
--turn-vp-reward-scale 0.01
--nuke-death-penalty 1.0
--high-stability-coup-penalty 0.03
--low-stability-warzone-coup-reward 0.03
--headline-opponent-event-penalty 0.03
--defcon-risk-pick-penalty 0.05
--defcon-risk-commit-penalty 0.15
--empty-country-influence-reward 0.01
--control-battleground-reward 0.02
--control-non-battleground-reward 0.005
--max-episode-step-penalty 0.25
```

`--reward-shaping-scale` multiplies VP/control/coup/headline/empty-country and
DEFCON-risk shaping. It does not scale terminal win/loss, the nuclear-loss
penalty, or the max-step truncation penalty.
For phase-out experiments, use `--reward-shaping-final-scale`,
`--reward-shaping-phaseout-start-episodes`, and
`--reward-shaping-phaseout-end-episodes`.

## Logging And Evaluation

Game logs:

- Full sampled game logs: `--log-games-dir`, controlled by `--log-games-every`.
- Terminal metrics: `--metrics-dir`; every terminal game writes compact JSONL metrics for W&B aggregation.

W&B metrics added for future runs:

- Rollout USSR/US/tie win rates.
- Rollout nuclear terminal rate.
- VP-threshold/final-scoring rates.
- Average game length.
- Mean episode reward components.
- Periodic checkpoint paths.
- Evaluation win rates versus:
  - the initial restored checkpoint;
  - random legal policy.

Periodic checkpoint/eval:

- Controlled by `--checkpoint-eval-every-episodes`, default `1000`.
- `--eval-games` now controls games per side/opponent pair, default `50`.
- `--eval-min-games-per-side` enforces a floor, default `10`. With
  `--eval-games 10`, each checkpoint plays 10 current-US games against the
  opponent USSR policy and 10 current-USSR games against the opponent US policy.
  Because the trainer evaluates both `initial` and `random`, that is at least
  40 total games per checkpoint eval before adding historical opponents.
- `--eval-history-opponents N` adds the most recent `N` evaluated checkpoints
  as benchmark opponents. Total games per eval are:
  `2 sides * max(eval_games, eval_min_games_per_side) * (2 + history_opponents)`.
  These historical comparisons are what make the Elo file a rolling benchmark
  instead of only an anchor check against `initial` and `random`.
- The trainer writes a persistent `elo_ratings.json` by default under the run's
  checkpoint directory. Override with `--elo-path`.
- Elo is side-specific:
  - `leaderboards.us` rates current US policies against USSR-side benchmark
    opponents such as `initial:ussr_policy`, `random:ussr_policy`, and recent
    checkpoint USSR policies.
  - `leaderboards.ussr` rates current USSR policies against US-side benchmark
    opponents such as `initial:us_policy`, `random:us_policy`, and recent
    checkpoint US policies.
- Checkpoint logs include scalar Elo fields such as
  `elo/us/current_rating`, `elo/us/current_games`,
  `elo/ussr/current_rating`, and `elo/ussr/current_games`.
- Long checkpoint evals emit `checkpoint_eval_progress` every
  `--eval-progress-every-games` completed games, default `10`, and
  `checkpoint_eval_opponent_done` after each opponent block. This is mainly
  useful when `--eval-games` is raised to 50+.
- Final checkpoint also runs evaluation.

## Known Limitations

- The default architecture is still feed-forward. `--model-arch transformer_history`
  adds a transformer over the fixed-length public action history, but hidden-hand
  belief is still learned indirectly from public zones, hand counts, and history.
- Region scoring features are approximate aggregates, not a full hand-crafted scoring oracle.
- Evaluation currently uses local policy weight swapping for current-vs-initial matches. This is lightweight and now maintains side-specific Elo ledgers, but it should still be treated as a trainer utility rather than a full tournament server.
- Checkpoint/eval cadence is episode-based in the current implementation, not raw environment-step-based, because full-game evaluation every 1k raw decision steps would dominate training time.
